# gen-prd

Generate a comprehensive, production-grade Product Requirement Document (PRD) by synthesizing all available product context in the workspace.

## How It Works

This skill reads from multiple sources — the PM role file (`pm-role.md`), project spec documents (`spec/`), debate outputs, materials, and source code — and produces a single structured PRD that an engineering team can directly implement from. It cross-references these sources to fill gaps, resolve conflicts, and produce requirements that are grounded in both product vision and technical reality.

## Core Principle

The PRD must be **specific, unambiguous, and decision-complete**. Every requirement must be concrete enough for an engineer to estimate effort and build without asking clarifying questions. Vague aspirations like "improve user experience" are forbidden.

## Prerequisites

**STOP** if none of the following sources exist in the workspace:

1. `pm-role.md` in the workspace root (generated by `/init-pm`)
2. A recognizable source codebase

If none are present, respond:

```
I need product context to generate a PRD. Please run /init-pm first,
or provide product materials (docs, user stories, competitive analysis).
```

Then **STOP** and wait for the user.

## Input Sources (Priority Order)

Read all available sources. When sources conflict, higher priority wins.

| Priority | Source | What It Provides |
|----------|--------|------------------|
| 1 | `pm-role.md` | Product vision, target audience, core features, business goals, constraints |
| 2 | `spec/` folder | Detailed architecture, module descriptions, data models, integrations (if generated by `/init-pm` from source code) |
| 3 | Source code | Implemented features, technical constraints, API surfaces, data models |

**Project Spec:** If a `spec/` folder exists in the workspace root, read `spec/README.md` first for navigation, then consult the relevant spec documents for accurate module descriptions, data models, integration details, and architecture context. The spec provides deep technical detail that the concise PM role file does not cover.

### Source Code Analysis

When source code is the primary or supplementary source:

1. **Check for project spec** — if `spec/` exists, read `spec/README.md` and relevant spec files first. The spec contains pre-analyzed architecture, module, and data model details that save effort and improve accuracy.
2. **Read documentation** — `README.md`, `CHANGELOG.md`, `docs/`, `CONTRIBUTING.md`
3. **Read manifests** — extract dependencies, scripts, version, description
4. **Scan directory structure** — infer features from folder names, module organization
5. **Read key source files** — API routes, schema definitions, feature modules, config
6. **Inspect tests** — test names reveal intended user flows and feature boundaries
7. **Inspect migrations/schemas** — data model and entity relationships

From code, extract:
- Existing features and their completeness
- Technical constraints (framework, language, database, deployment)
- API surface area and integration points
- Data model and entity relationships
- Authentication/authorization patterns
- Performance characteristics (caching, indexing, query patterns)

Save the generated PRD to:


- Create the `prd-outputs/` directory if it does not exist.
- The PRD content language follows the PM role file language; default to **Simplified Chinese** if no language preference is set.

## PRD Structure (Mandatory)

The PRD must contain **all** of the following sections. Do not skip any section — mark "N/A" with a brief explanation if a section truly does not apply.

### 1. Document Metadata

```markdown
# <Product Name> — Product Requirement Document

| Field | Value |
|-------|-------|
| Version | 1.0 |
| Date | <YYYY-MM-DD> |
| Author | AI Product Manager |
| Status | Draft |
| Source | <list of input sources used> |
```

### 2. Executive Summary

3-5 sentences covering:
- What the product is
- Who it serves
- The core problem it solves
- Why now (timing / market urgency)
- The key differentiator

### 3. Problem Statement

- **User pain points** — specific, observable problems with evidence from materials
- **Market gap** — what existing solutions fail to address
- **Cost of inaction** — what happens if this problem is not solved

### 4. Product Vision & Goals

- **Vision statement** — 1-2 sentences, aspirational but grounded
- **Goals** — 3-5 measurable goals with success criteria
- **Non-goals** — explicit exclusions to prevent scope creep

### 5. Target Users & Personas

For each persona (minimum 2, maximum 5):

| Field | Description |
|-------|-------------|
| Name | Fictional but descriptive name |
| Role | Job title or life context |
| Primary need | The #1 problem this persona faces |
| Current workaround | How they solve it today |
| Success looks like | What changed after using the product |
| Frequency | How often they use the product |

### 6. User Stories & Scenarios

For each major feature area, provide:

```markdown
#### US-<ID>: <Title>

**As a** <persona>,
**I want to** <action>,
**so that** <benefit>.

**Acceptance Criteria:**
- [ ] Criterion 1 (specific, testable)
- [ ] Criterion 2
- [ ] Criterion 3

**Edge Cases:**
- What happens when <boundary condition>?
- What happens when <error scenario>?

**Priority:** P0 / P1 / P2
**Complexity:** High / Medium / Low
```

### 7. Functional Requirements

Organize by feature area. For each requirement:

| Field | Description |
|-------|-------------|
| FR-ID | Unique identifier (FR-001, FR-002, ...) |
| Feature Area | Which feature this belongs to |
| Description | What the system must do |
| Input | What data or trigger initiates this |
| Output | What the system produces or changes |
| Priority | P0 / P1 / P2 |
| Dependencies | Other requirements this depends on |

### 8. Non-Functional Requirements

| Category | Requirement | Target |
|----------|-------------|--------|
| Performance | Page load time | < 2s (LCP) |
| Performance | API response time | < 200ms (p95) |
| Availability | Uptime | 99.9% |
| Scalability | Concurrent users | <define based on context> |
| Security | Authentication | <method> |
| Security | Data encryption | At rest + in transit |
| Accessibility | WCAG compliance | AA |
| Internationalization | Supported languages | <list> |
| Browser support | Minimum browsers | Chrome, Firefox, Safari, Edge (latest 2) |

### 9. Data Model

- **Entities** — list all core entities with attributes
- **Relationships** — describe relationships (1:1, 1:N, M:N)
- **Key constraints** — uniqueness, required fields, enum values
- **Data lifecycle** — creation, updates, archival, deletion policies

Present as text description or simple ER-style notation.

### 10. API Design (if applicable)

For each endpoint or integration point:

```
<METHOD> <path>

Purpose: <what it does>
Auth: <required>
Request: <payload shape>
Response: <payload shape>
Errors: <possible error codes>
```

### 11. UI/UX Requirements

- **Key screens** — list the primary screens/views with their purpose
- **User flow** — describe the critical path from entry to value delivery
- **Interaction patterns** — specify any required patterns (real-time updates, drag-and-drop, etc.)
- **Responsive requirements** — breakpoints and behavior
- **Accessibility requirements** — keyboard navigation, screen reader support, color contrast

### 12. Dependencies & Integrations

| Dependency | Type | Purpose | Fallback |
|------------|------|---------|----------|
| <service> | External API / Library / Service | Why needed | What if unavailable |

### 13. Release Strategy

- **MVP scope** — the minimum set that delivers core value
- **Phase 2** — features that enhance but are not blocking
- **Phase 3** — future considerations
- **Feature flags** — which features should be behind flags
- **Rollout plan** — percentage rollout, canary, or big-bang

### 14. Success Metrics

| Metric | Definition | Target | Measurement Method |
|--------|------------|--------|--------------------|
| <metric> | How it's calculated | Goal | How to track |

Include at least:
- 1 acquisition metric
- 1 engagement metric
- 1 retention metric
- 1 business outcome metric

### 15. Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| <risk> | High / Medium / Low | High / Medium / Low | <action> |

Include at least 5 risks covering: technical, market, resource, timeline, and dependency risks.

### 16. Open Questions

Questions that must be answered before or during development:

| Question | Owner | Deadline | Status |
|----------|-------|----------|--------|
| <question> | <who> | <when> | Open |

### 17. Appendix

- Glossary of domain terms
- Links to source materials
- Reference to debate outputs (if any)
- Competitive analysis summary
- Technical architecture diagrams (text-based)

## Quality Standards

### Specificity

Every requirement must pass the "engineer test": can an engineer estimate and build this without a follow-up meeting? If not, it is too vague.

| Vague (forbidden) | Specific (required) |
|--------------------|---------------------|
| "Fast loading" | "LCP < 2.5s on 3G connection" |
| "User-friendly" | "Complete primary flow in <= 3 clicks" |
| "Secure" | "OAuth 2.0 + PKCE, JWT with 15min expiry, refresh token rotation" |
| "Scalable" | "Support 10K concurrent users with p95 < 200ms" |

### Consistency

- Terms must be used consistently throughout (define in glossary)
- Priorities must not conflict between sections
- User stories must trace back to functional requirements
- Success metrics must align with goals

### Completeness

- Every P0 feature must have user stories, functional requirements, and acceptance criteria
- Every entity in the data model must appear in at least one user story
- Every external dependency must have a fallback strategy
- Every risk must have a mitigation

## Anti-Patterns

| Mistake | Fix |
|---------|-----|
| Writing vague aspirations instead of concrete requirements | Every requirement must be testable and measurable |
| Skipping non-functional requirements | Performance, security, and accessibility are not optional |
| Ignoring edge cases | List at least 2 edge cases per user story |
| No success metrics | Every goal needs a measurable outcome |
| Copying pm-role.md verbatim | Synthesize and expand; the PRD is 10x more detailed than the role file |
| Ignoring debate outcomes | If debates happened, their decisions must be reflected in requirements |
| Missing open questions | Flag unknowns explicitly; do not hide ambiguity in vague requirements |
| No prioritization | Every requirement needs P0/P1/P2; not everything can be P0 |
| No release strategy | Define MVP vs. future phases to prevent scope creep |
| Treating source code features as requirements | Code shows what was built, not necessarily what should be built — validate against product goals |
